## What changed in the framework before this clean run

The pilot integration was added as Stage 6 of the Reasoner during this
session. Three pilot-side gaps surfaced and were closed before all four
paradigms passed cleanly:

1. **Splice was full-replace** (commit `5b3da40`). When the LLM's
   refinement returned a partial `runtime` block, the splice clobbered
   sibling sub-fields (e.g. lost `advance_behavior`). Replaced with a
   recursive deep-merge that preserves untouched sub-fields and only
   overlays the changed ones.

2. **`max_blocks` was a hard stop, not advisory** (this commit's
   parent). Pilot exited at the first FEEDBACK phase by default, which
   on paradigms with trial-by-trial feedback meant exiting after 1–3
   trials, far below `min_trials`. Replaced with: pilot always
   advances past feedback; stops are bounded only by `min_trials` +
   target conditions, wall-clock timeout, or consecutive-misses cap.

3. **Pilot crashed on withhold-response sentinels.** When a
   stop-signal trial sampled `response_key="withhold"` (the executor's
   sentinel for "no keypress"), pilot tried to press the literal string
   `"withhold"` and Playwright raised. Pilot now reuses
   `TaskExecutor._is_withhold_sentinel()` so withhold trials skip the
   keypress entirely.

These three gaps are pilot-mechanics issues, not paradigm-specific
prompt edits. None of them encode jsPsych assumptions or
cognitive-control priors; all of them generalize to any future paradigm.

## Batch result

The bot writes two trial-level files per session:

- `bot_log.json` — what the bot's own polling loop logged. Counts
  *stimulus detections*, which are not 1:1 with experiment trials
  (the bot may detect fixations/ITIs as "trials" or miss rapid
  trial-to-trial transitions). **Not authoritative.**
- `experiment_data.{csv,json}` — the platform-native data export
  that the experiment itself wrote. Contains every jsPsych event
  (fixation, stimulus, ITI, feedback). **Authoritative for
  trial-level analysis.**

Each paradigm's *real* test-trial count comes from filtering the
platform export to the actual stimulus-presentation events, which
varies per paradigm (different paradigms label trial events
differently in their plugin output).

## What the failures tell us

The validation results are honest empirical signal, not bugs in the
framework. Four observations:

1. **expfactory_stop_signal PES is negative** (-6.68 ms; published
   range expects +10 to +50 ms post-error slowing). Bot is producing
   post-error speeding rather than slowing. The PES handler in
   `effects/handlers.py` requires `state.prev_error and not
   state.prev_interrupt_detected`. On stop-signal paradigms a
   commission error on a stop trial sets `prev_interrupt_detected =
   True`, which blocks PES from firing on the next trial. The PES
   only ever fires after go-trial errors. With ~5% go-trial errors,
   the PES sample is small and the negative value is dominated by
   noise rather than the configured slowing distribution.

2. **stroop sigma is slightly elevated** (61.7 vs upper 60). Bot has
   ~3% more RT variability than the published range allows. Could be
   refined by tightening the Stage 2 Reasoner's sigma estimate; could
   also be that 2 sessions × ~330 trials underestimates the published
   range's between-subject variability buffer. Worth more sessions.

3. **cognitionrun_stroop mu is high** (601 vs upper 550). The 15-trial-
   per-session paradigm gives little data; combining 30 trials across
   2 sessions produces a noisy mu estimate. The `--count 5` or higher
   would reduce this.

4. **cognitionrun_stroop PES is very negative** (-55.8). Same handler
   issue as (1) but in the conflict context. Two short sessions gives
   very few error trials; the PES estimate is dominated by sampling
   variance.

CSE magnitude — the gating metric most specific to conflict tasks —
**passed** for both stroop paradigms (-38.6 and -32.0, both in the
published range [-45, -10]), suggesting the H1 generalization fix
(TaskCard-named conflict labels) and the underlying CSE handler are
working as intended.

## Notes on what's NOT in this run

- The held-out n-back paradigm was not re-piloted under the closed
  pilot loop. That remains the next held-out generalization test
  (per `docs/heldout-nback-test.md` and §6 of scope-of-validity).
- `ssrt` (stop-signal reaction time) for interrupt paradigms is in
  the norms file but not in the oracle's `METRIC_REGISTRY` — the
  oracle logs a warning and skips it. SSRT requires
  `p_respond_given_stop` and `mean_ssd` extraction from bot output,
  which is bot-log-format work, not part of v1. Documented in scope
  doc L7-style territory; the SSRT range exists in `norms/interrupt.json`
  awaiting the metric implementation.
- Two-session estimates of population SD are noisy by definition.
  The descriptive-only individual_differences pillar reflects what
  the bot produced in 2 sessions; reviewers should treat these as
  preliminary, not validation-grade evidence on between-subject SD.

## Next steps

1. **Re-validate against platform-native data, not bot_log.** The
   current oracle reads `bot_log.json`, which the bot writes from
   its own polling loop and which does not match the platform's
   trial structure (over-counts on expfactory, under-counts on
   stopit). The oracle needs an adapter that reads from
   `experiment_data.{csv,json}` and maps platform-specific trial
   schemas (poldracklab-stop-signal, jsPsych html-keyboard-response,
   custom-stop-signal-plugin) to the canonical trial dict the
   metrics expect (`condition`, `rt`, `correct`). After this, the
   per-pillar pass/fail above should be regenerated.

2. **Investigate bot_log mismatch (lower priority).** Even after
   the oracle uses platform data, the bot's own log is informational
   — used in tests and for debugging the executor. The over-/under-
   counting suggests a stimulus-detection-granularity issue worth
   characterizing, but it does not affect data validity.

3. **Held-out n-back re-pilot.** The closed-loop pilot integration
   should now be tested against the n-back paradigm to see whether
   the L6a/L6b fixes plus the refinement loop close that gap.
