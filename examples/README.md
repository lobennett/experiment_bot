# Example Output

Representative output from a single bot run of each validated task. These files illustrate what the bot produces and how the pipeline's components connect.

## Directory Structure

```
examples/
├── expfactory_stop_signal/     # ExpFactory RDoC Stop Signal
├── expfactory_stroop/          # ExpFactory RDoC Stroop
└── stopit_stop_signal/         # STOP-IT Stop Signal
```

Each directory contains 4 files produced by a single bot run.

## Files

### `config.json`

The TaskConfig used for this run. Generated by Claude from the experiment's source code, validated by a pilot run, then cached. Contains:

- **`task`** — Task name, cognitive constructs, and literature references identified by Claude
- **`stimuli`** — Detection selectors and response mappings for each stimulus type. These are JavaScript expressions or CSS selectors that the bot evaluates on every poll cycle to identify what's on screen
- **`response_distributions`** — Ex-Gaussian (mu, sigma, tau) parameters for each condition's reaction time distribution, inferred by Claude from the cognitive psychology literature
- **`performance`** — Per-condition accuracy and omission rate targets, also literature-derived
- **`temporal_effects`** — Sequential dependencies in the RT series (autocorrelation, fatigue drift, post-error slowing, etc.). Each effect includes a `rationale` field citing the literature basis. All parameters determined by Claude
- **`between_subject_jitter`** — How much each simulated session varies from the base parameters, representing individual differences
- **`pilot`** — Parameters Claude specified for the pilot validation run (min trials, target conditions)
- **`navigation`** — Steps to get from the experiment's landing page to the first trial (button clicks, keypresses, waits)
- **`runtime`** — Phase detection expressions, timing parameters, advance behavior, data capture method

This file is the complete specification of the bot's behavior. The Python code contains no behavioral assumptions — everything comes from this config.

### `bot_log.json`

Per-trial decision log recording what the bot did on each trial:

- `trial` — Trial number
- `stimulus_id` — Which stimulus was detected (matches a `stimuli[].id` from config)
- `condition` — Experimental condition label
- `response_key` — Key the bot pressed (or null for omissions/withheld responses)
- `sampled_rt_ms` — Reaction time sampled from the ex-Gaussian distribution (with temporal effects applied)
- `actual_rt_ms` — Actual elapsed time from stimulus onset to keypress (includes system latency)
- `omission` — Whether the bot intentionally omitted a response
- `intended_error` — Whether the bot intentionally pressed the wrong key
- `rt_distribution` — Which distribution was sampled from (e.g., `go`, `congruent`)

For stop-signal tasks, stop trials appear with conditions like `stop_withheld` (successful inhibition) or `stop_responded` (failed inhibition).

### `experiment_data.{csv,json}`

Raw data captured from the experiment platform after task completion. This is the experiment's own data — not the bot's decision log. The format and columns depend on the platform:

- **ExpFactory (jsPsych 7)** — JSON or CSV with columns like `rt`, `response`, `correct_trial`, `trial_id`, `condition`, `SSD` (for stop signal). Includes practice trials, instruction trials, and metadata rows alongside test trials.
- **STOP-IT** — CSV with columns like `rt`, `response`, `correct`, `signal`, `SSD`. All rows are test trials.
- **Cognition.run** — CSV with columns like `rt`, `response`, `text`, `colour`. Requires post-hoc condition derivation.

The analysis notebook (`scripts/analysis.ipynb`) filters these to test trials and computes metrics from the experiment's own correctness scoring, not from the bot's `bot_log.json`.

### `run_metadata.json`

Run metadata:

- `task_name` — Name from the TaskConfig
- `task_url` — Experiment URL
- `total_trials` — Number of trials the bot completed
- `headless` — Whether the browser was visible

### `screenshots/` (not included)

Error screenshots are captured when the bot encounters an unhandled exception during execution. These are useful for debugging selector failures or navigation issues. Not present in these examples because the runs completed successfully.
