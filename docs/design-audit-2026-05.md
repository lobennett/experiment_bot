# Codebase Design Audit — experiment_bot

_Multi-agent audit (43 agents): 7 subsystems mapped, 69 findings, 67 confirmed, 2 refuted after adversarial verification. Generated 2026-05-28._

## Executive summary

The framework's scientific core is sound and well-defended: the generic-mechanism effect library, the open paradigm-class taxonomy, the deterministic Claude-free oracle, and the structural separation of the parameter-setting Reasoner (primary studies) from the gating norms (meta-analyses) are correctly built and should be preserved exactly (claude-006, domgen-007, humanlike-008, platform-006, arch-014, defensibility-008, robust-010). Where the design is healthy, it is genuinely healthy. The flaws cluster almost entirely in the STRUCTURAL layer (navigation, selector/key resolution, completeness, calibration, per-paradigm adapters) and in CONTRACT DRIFT (five overlapping TaskCard definitions; docs advertising capabilities the code does not honor). The single deepest issue is that navigation is modeled as an offline-frozen TaskCard artifact while real entry flows are interactive — SP13-16 are four patches against this one wrong abstraction (genbottle-001). The single highest-leverage, lowest-risk lever, however, is not that XL redesign: it is closing the data-integrity hole whereby any session that reaches >=1 trial but terminates early is scored as if complete (robust-001, high severity, confirmed), because that directly endangers the reviewer-defensibility the whole project stakes itself on, generalizes to every paradigm, and is M-effort/low-risk. Several "advertised but discarded" capabilities (RT distribution family choice; documented working_memory gating metrics that do not exist; empty provenance hashes; stale model id) are honest-defensibility liabilities that are cheap to fix and should be done first as a coherence pass. The leverage-Claude tenet is satisfied at TaskCard-generation time (Stage 1 selector/nav reasoning, Stage 2 behavioral parameters); the documented SP9a negative result correctly closes the door on a runtime LLM key-resolver — that absence is evidence-backed, not an oversight.

## Fundamental flaws (warrant redesign, not patch)

### Navigation is modeled as an offline-frozen artifact, but real entry flows are interactive — the TaskCard-static-nav abstraction is the wrong primitive

The single held-out paradigm required FOUR sub-projects (SP13 sequential walker -> SP14 nav schema in prompt -> SP15 platform defaults + persistent walker -> SP16 runtime adaptive nav), each patching a symptom of one root cause: navigation cannot be fully decided offline from static source because cognitive-experiment entry flows interleave instructions and demo trials and depend on runtime DOM state. The SP15 walker still bakes trial-response keypresses into navigation.phases (verified: stop_signal_with_integrated_memory/f6772248.json phases 11-13 are '.'/','/'.' trial responses), producing TaskCards the executor cannot replay; the executor then needs its own runtime LLM nav that burned 10/10 budget on the held-out paradigm. SP16's adaptive nav IS the actual fix (Claude-in-the-loop against the live DOM) but it is bolted on as a recovery mechanism gated to one failure mode while the frozen static-nav remains the authoritative primary engine, and a Stage-6 PASS still does not imply an executor-replayable card. This is the limiting abstraction for the url2task end-goal on any paradigm with a non-canonical entry flow. CLAUDE.md SP15 calls the mismatch 'an architectural limit of the executor' and scope L1 documents static-source-only nav, but neither names the offline-frozen-nav abstraction itself as the root bottleneck. Note: this caps NAVIGATION generalization only; the behavioral-fidelity scientific core is orthogonal and unaffected, and SP16 did capture 666 trials across all 5 conditions, so the framework is not blocked — it generalizes via accreted machinery rather than a clean primitive.

_Findings: genbottle-001, genbottle-007, arch-005, platform-001, claude-003, genbottle-004, platform-002_

### Per-trial response correctness is at chance for content-dependent paradigms, and this propagates into the platform-scored accuracy/PES gates without being documented as a ceiling

The rubric names 'which key to press' as a first-class leverage-Claude decision, but runtime key resolution is a purely static 4-tier chain (no LLM), and CLAUDE.md SP8 records it stays at chance for stroop/stop_signal_expfactory/stop-it. This is NOT an oversight — SP9a built and tested a runtime LLM key-resolver and found it empirically NOT supported (the 'one key per condition' abstraction was structurally inadequate; the deeper cause is an information deficit: the page often lacks a usable correct-response variable), and SP12 removed it. So the absence is deliberate and evidence-backed. The fundamental aspect is the bounded ceiling it creates plus the under-documentation of its consequence: for a held-out paradigm whose correct response depends on stimulus content AND whose platform scores against the true mapping, chance-level keys corrupt the platform-adapter-derived `correct` field, which corrupts measured accuracy and the post_error_slowing gate partition. cse_magnitude, rt_distribution, lag1_autocorr, between_subject_sd, and ssrt are all correctness-INDEPENDENT and therefore immune (verified in the oracle), so the practical cap is narrow (accuracy-type gates, i.e. working_memory's n_back_accuracy) and the conflict/interrupt classes the project actually targets are unaffected. The right response is documentation (an explicit scope ceiling), not re-litigating the SP9a runtime-LLM dead-end.

_Findings: claude-001, genbottle-003, robust-003_

## Themes

### Data-integrity / hard-fail gaps that silently produce falsely-scientific numbers

G4's 'hard-fail on broken state' is implemented as a single trial_count==0 check. Any run reaching >=1 trial but terminating early (window desync, too-many-misses, false-positive COMPLETE) writes a structurally valid bot_log, exits 0, and is scored by the oracle as if whole — and a313e9f (61 vs 124 trials on stroop) shows this already happened. Compounding: pervasive broad except-Exception blocks mask Reasoner-emitted-JS bugs as benign no-ops, the OutputWriter silently no-ops saves when create_run was never called, and a false-positive COMPLETE (exception-as-done heuristic) feeds the same silent-partial hole. These convert configuration/navigation failures into plausible-but-wrong gating metrics rather than loud failures, directly threatening reviewer defensibility.

_Findings: robust-001, robust-002, robust-005, robust-009, robust-008_

### Advertised-but-discarded capability: docs and prompts claim more than the code honors (broken leverage + honesty liabilities)

Multiple capabilities are computed and thrown away or documented as working when they are not. The Reasoner's RT distribution-family choice (ex_gaussian/lognormal/shifted_wald) is dropped at the TaskCard boundary and hardcoded to ex_gaussian; a non-ex_gaussian choice would crash mid-session (KeyError); scope L4 and a sampler docstring assert all three families are selectable. scope §5 advertises working_memory gating metrics (n_back_accuracy_2back, capacity_k) that exist in neither the norms file nor the oracle (trimmed in ecf07ea, docs never updated), and the all-descriptive-class -> overall_pass=False collision means a perfectly humanlike n-back session reports FAIL. Stage 4 DOI verification gates nothing and crashes on malformed citations. These erode the paper-trail defensibility the project stakes itself on, all at low-to-medium effort.

_Findings: claude-002, domgen-001, humanlike-003, arch-002, domgen-003, humanlike-007, defensibility-002, genbottle-005, defensibility-006_

### Contract drift: five overlapping TaskCard definitions and dead/half-migrated code with no single source of truth

The TaskCard contract is independently defined in schema.json, config.py from_dict/to_dict, taskcard/types.py, normalize.py, and validate.py, and they disagree: schema.json still advertises deprecated condition_repetition and hurst (a committed dev card still enables condition_repetition and fires the deprecation warning), two registered effects bypass shape validation (6-vs-8 gap), and ~150 LOC of dead Stage-6 refiner plus a dead analyzer.py plus a vestigial StuckDetector and an always-empty pilot_validation field survive only because tests pin them — giving false confidence that the live path is what the tests assert. This is the accretion the audit targets and it directly hurts generation quality (Claude is still steered to deprecated fields via system.md).

_Findings: arch-001, arch-006, arch-010, arch-012, arch-013, claude-008, humanlike-006_

### Anti-circularity erodes silently on exactly the held-out case the generalization story depends on

When no platform adapter is registered (the default state for any held-out paradigm), the oracle soft-warns (stderr only, no hard fail) and falls back to scoring against the bot's OWN self-reported intended_error — the bot grading its own homework. Only post_error_slowing actually consumes `correct`, and SP16's held-out run was scored manually from the authoritative export (per G4 discipline), so the hole is currently latent and bounded — but it is undocumented as a circularity hole (L8 frames it benignly as a counting risk) and nothing in the committed report records which loader ran. Related: cse_magnitude's gate EXISTENCE depends on a bot-side artifact (the TaskCard modulation_table), so a misconfigured card makes a published-range gate silently vanish (NaN->non-gating) rather than fail loudly.

_Findings: robust-004, defensibility-004, platform-004, defensibility-005, genbottle-006_

### Code-side numeric/method priors fill behavioral fields the Reasoner is supposed to own

normalize.py silently injects accuracy={'default':0.95} when Claude omits accuracy — a concrete behavioral prior set by Python, not the literature, which can make a low-accuracy paradigm score as 95% accurate; it also defaults detection.method to js_eval silently, and system.md:251 carries a soft editorial RT-calibration nudge. These are precisely the 'work around Claude in Python' shims the no-numeric-priors stance (R5/R6, G2) forbids, even though those rules name prompts specifically. Cheap to fix by re-prompting or surfacing an auditable sentinel.

_Findings: claude-007, domgen-005_

### Calibration measures the wrong quantity and is currently inert; SSRT-high is a delivery artifact mis-attributed to the Reasoner

The calibration pass fits its offset on a fixed deliverer dwell (50/225ms) and applies it to full sampled trial RTs delivered through a different (external-sleep, dwell=0) pathway, and a constant-interval override collapses the regression fallback to fixed-offset. But the flaw is currently latent: every recorded run lands on too_few_events (0/30 correctly recorded, the SP7 layer-d platform-recording gap), so the adjustment is a no-op and has never been applied. The systemic SSRT-high across stop-signal paradigms is the same delivery-pipeline inflation manifesting in the interrupt class — a downstream consequence of delivery timing, not the Reasoner mis-parameterization the SP16 docs attribute it to. The honest framing in the docs is good; only the causal attribution needs correcting. Calibration also calibrates on Space because the response-key set is empty at calibration time.

_Findings: humanlike-002, humanlike-005, robust-007_

### Reproducibility, provenance, and model-leverage gaps weaken the G4 paper trail

SP16 adaptive nav puts a nondeterministic, seedless LLM call in the trial-capture path, so a session is no longer reproducible from (seed + taskcard_sha256) — documented in the SP16 spec but NOT in the reviewer-facing scope-of-validity, and the in-code 'same seed = same output' comment is now false. Provenance hashes (prompt_sha256, source_sha256) are emitted empty and the model id is a stale hardcoded 'claude-opus-4-7' across five sites while the harness runs opus-4-8, so produced_by can attribute a card to a model that never produced it. load_latest selects the running card purely by filesystem mtime. There is no --model flag, no per-stage tiering, and no prompt caching despite re-sending large identical prefixes every stage.

_Findings: defensibility-001, arch-007, arch-011, claude-005, defensibility-003_

### Domain-generality ceilings that are documented but where the stated target set slightly exceeds the model

The trial model is hardwired to single discrete keyboard response per stimulus (no continuous/mouse/multi-response, no study-probe/cued/multi-stimulus trial structure). This is an explicit, deliberate non-claim (scope §3, keyboard-only). The genuine residual is narrow internal-doc inconsistency: CLAUDE.md G1 names WCST (mouse sort) and scope §6 lists Sternberg (study-probe) as targets that the keyboard-only single-response model cannot express. Also: error trials draw RT from the correct-trial distribution and pick a uniformly-random wrong key (no error-RT speedup), a latent fidelity gap the executor already has machinery (_error variants) for. The SSRT oracle hardcodes 'go'/'stop' condition labels, requiring a Python edit to score a novel interrupt paradigm.

_Findings: domgen-002, domgen-006, humanlike-004, domgen-004_

### God-class concentration and dual-engine navigation inconsistency

TaskExecutor is a 1335-LOC class mixing seven responsibilities with a large mutable-state surface, which is precisely why three distinct bugs in this audit (calibration ordering, nav-engine divergence, response-window triplication) co-locate here and interact through shared state. Two divergent navigation engines (PilotSession.try_phase vs InstructionNavigator.execute_all) support different action sets, so a 'repeat' phase present on a committed dev card is silently dropped at entry but honored on re-run — the same TaskCard has path-dependent semantics. The executor also imports a private reasoner helper (the one genuinely inverted layering edge, though documented and harmless).

_Findings: arch-008, arch-009, platform-003, robust-006, arch-003_

### Roadmap rationale

**#1 — Close the completeness/data-integrity hole: persist a trial-loop exit-reason enum (complete / window_closed / max_misses / budget / context_destroyed) into run_metadata, set metadata['incomplete'] when exit != complete, and have the oracle EXCLUDE incomplete sessions from aggregation while recording them in the report (N_used vs N_supplied with reasons). Keep the existing ==0 hard-fail. Do NOT hard-raise on early-break by default (would break legitimate held-out exploration) and treat any expected_trial_count as advisory only.**

This is the single highest impact-per-effort change. It directly protects the reviewer-defensibility the entire project stakes itself on, generalizes to every paradigm (no paradigm vocabulary), and closes a confirmed already-occurred failure (61 vs 124 on stroop) that the only existing guard did not catch. It also becomes the substrate that robust-002, robust-005, robust-007, and robust-008 hook into. M-effort, low-risk, and it builds on the existing well-designed run_trace instrumentation rather than replacing it.

**#2 — Run a single coherence/honesty pass that aligns docs+code+prompts (no new capability): (a) carry the distribution family end-to-end OR delete the lognormal/shifted_wald samplers and the Stage 2 family-choice ask, and fix scope L4 + the sampler docstring either way; (b) rewrite scope §5 working_memory to say it has NO gating metrics (note the ecf07ea trim) and make the all-descriptive class return overall_pass=None ('unscored'), surfaced distinctly in cli output, instead of a misleading False; (c) populate real prompt_sha256/source_sha256 and source the model id from the live client (one DEFAULT_MODEL constant); (d) surface doi_verified counts and use .get() in Stage 4 to stop the KeyError crash.**

These are all low-risk, low/medium-effort fixes to capabilities the docs ADVERTISE but the code does not honor — exactly the liabilities that would embarrass the framework under reviewer scrutiny. They share a theme (G5 same-pass discipline: doc and code must agree) and several touch the same files, so batching is efficient. None weakens any gate; all strengthen the paper trail. Crash-prevention (distribution KeyError, Stage 4 KeyError) and the working_memory always-FAIL are the most urgent within this bundle.

**#3 — Harden anti-circularity on the held-out path: make a missing adapter a HARD-FAIL for a gating run (or require an explicit --allow-bot-log flag that stamps the report JSON with data_source='bot_log_self_graded'); when the bot_log fallback is used, mark correctness-dependent metrics (post_error_slowing) as descriptive-only so the self-graded `correct` can never gate; escalate the warning to name the circularity; and have data_capture distinguish 'no method configured' from a swallowed exception in run_metadata.**

This closes the one place where G4's central anti-circularity guarantee silently degrades — on exactly the held-out case the generalization story depends on. It is currently latent (SP16 was scored manually per G4), but it is enforcement-by-discipline rather than by code, and nothing in the committed artifact records that the bot graded itself. S-to-L effort for the load-bearing report-marker + correctness-metric guard; defer the full TaskCard field-mapping adapter migration (L8) as the longer-term generalization answer.

**#4 — Finish the contract-drift migration and prune dead code in one G5 pass: remove condition_repetition and hurst from schema.json + system.md + the Stage 2 registry injection (keep only from_dict deprecation shims for archived cards), regenerate the four dev cards so none fire deprecation warnings, add schema subschemas for practice_effect + vigilance_decrement (close the 6-vs-8 gap), add a test asserting schema temporal_effects.properties == EFFECT_REGISTRY handler keys, and delete the dead Stage-6 refiner, analyzer.py, StuckDetector, and the empty pilot_validation field (retarget their tests at the live paths).**

Removes the accretion that hurts generation quality (Claude is still steered to deprecated fields) and the dead code whose tests give false confidence about the live path. Medium effort, contained risk behind the existing suite, and it shrinks the five-way contract drift toward the single-source-of-truth the project wants. Defer schema-as-canonical autogeneration (legitimately a Tier-2/3 backlog item).

**#5 — Remove code-side behavioral priors and fix the calibration ordering bug: replace the silent accuracy=0.95 default with a re-prompt (feed the validator error) or an auditable run_metadata sentinel; make the js_eval method default warn; reword the system.md:251 RT nudge to a neutral derivation rule; and populate the calibration response-key set from the TaskCard key_map before the calibration pass (instead of the empty _seen_response_keys, which makes it always calibrate on Space).**

These are small, low-risk fixes that restore the no-numeric-priors discipline (G2/R5/R6) and the leverage-Claude tenet — the Reasoner, not Python, should own behavioral values. The calibration-on-Space ordering bug is an unintended defect (vs the documented escalate no-op tradeoff) that should be corrected before any calibration re-measurement.

**#6 — Document the honest ceilings as explicit scope-of-validity entries (no code): (a) per-trial key correctness can be at chance for content-dependent paradigms scored against the true mapping, propagating into accuracy + PES gates only (cse/rt/ssrt/lag1/SD immune), cite SP7/SP8/SP9a; (b) SP16 adaptive-nav sessions are not seed-reproducible (correct the in-code 'same seed = same output' comment too); (c) name the offline-frozen-nav abstraction as the known structural limit for interleaved instruction/trial entry flows; (d) reconcile CLAUDE.md G1 (drop WCST) and scope §6 (drop/annotate Sternberg) against the keyboard-only single-response trial model; (e) re-attribute systemic SSRT-high to the delivery pipeline, not the Reasoner.**

Cheap, high-trust-yield honesty work that converts implicit ceilings into explicit non-claims — directly aligned with the owner's honest-generalization-framing preference. It also prevents future effort being misdirected (e.g. re-deriving already-correct go-RT parameters to chase SSRT, or re-litigating the SP9a runtime-LLM dead-end).

**#7 — Unify navigation onto one engine and one Claude-driven policy: fix the Stage-6 walker to classify trial-response keypresses vs nav advances (gate appends on whether a trial stimulus was rendered/consumed) so walker cards are executor-replayable by construction; share one _propose_next_phase policy between Stage 6 and the executor; add 'repeat' (and any other action) to PilotSession.try_phase so entry-nav and re-run have identical semantics; keep adaptive nav gated on the stuck-DOM trigger (do NOT make it eager — that regression is proven). Demote platform_defaults backfill to platform-infrastructure-only phases (drop the cognition.run keypress-count) and change the clobber rule to append-missing-tail rather than replace-when-shorter; add a test pinning constants to source cards or derive them at runtime.**

This attacks the fundamental navigation abstraction at its real source (the walker conflating demo-trial responses with advances) without re-implementing the eager adaptive-primary engine that the SP16 first-cut proved regressive. It makes a Stage-6 PASS imply an executor-replayable card, eliminates the dual-engine 'repeat' divergence, and reduces the paradigm-overfitting hazard in platform_defaults. XL/high-risk overall, so it follows the cheaper integrity and honesty work; but it is the durable fix for the deepest generalization ceiling and should not be deferred indefinitely.

**#8 — Decompose TaskExecutor incrementally behind the test suite: extract a CalibrationCoordinator and a TrialRunner (owning the trial state as a dataclass, consolidating the three response-window polling paths into one timeout-configurable helper), keep run() as a thin orchestrator, narrow the broad except Exception blocks to PlaywrightError where the comment claims teardown so genuine JS-eval errors surface, and relocate the one inverted import edge (_propose_next_phase) into a neutral core module.**

Lowest priority because it is pure maintainability with the highest risk/effort and no direct goal impact, but it removes the structural reason the integrity bugs co-locate and interact, and narrowing the excepts amplifies the rank-1 diagnostic work. Do it last, incrementally, and only after the behavior-affecting fixes have landed.

## Keep as-is (sound choices; do not erode)

- The Claude-free deterministic oracle and the structural separation of the two evidence tiers (Reasoner cites primary studies to set parameters; norms-extractor cites ONLY meta-analyses to set gating ranges; neither runs at scoring time). This is the framework's strongest defensibility property. Resist any future suggestion to add a runtime LLM 'humanlikeness judge' at scoring time — it would reintroduce circularity and destroy reproducibility. (claude-006, defensibility-008)
- The generic-mechanism effect registry + open paradigm-class taxonomy + data-driven oracle dispatch: handlers name only mechanism vocabulary (autocorrelation, lag1_pair_modulation, post_event_slowing, pink_noise), read all paradigm specifics from cfg, and contribute zero when disabled; cse_magnitude is a thin named wrapper over the generic lag1_pair_contrast. New paradigm CLASSES can be added by Claude + a norms file with no library edit. Keep the negative-assertion tests that guard paradigm vocabulary out of the registry. (domgen-007, humanlike-008, genbottle-008)
- The framework-agnostic Stage-1 prompt, config-driven phase detection, and config-driven data capture — zero platform-specific selectors are hardcoded; jsPsych appears only as illustrative examples; the bot library is mechanism-only. Protect this with a negative test asserting no platform-host literals in system.md and no paradigm/platform names in phase_detection.py / data_capture.py; treat it as the target shape that platform_defaults.py and platform_adapters.py should migrate TOWARD. (platform-006)
- The behavioral-parameter generative model: Claude reads the literature and configures generic mechanisms with per-parameter citations + literature_range; the ex-Gaussian draw + additive sequential modulations reproduces correct QUALITATIVE trial-to-trial dynamics (Stroop CSE -34ms within norm, lag-1 autocorr 0.22 within norm, stop-signal race-model ordering holds). When fixing the delivery/calibration pipeline, leave the sampler and effect handlers untouched — they set the target RT correctly. (humanlike-008)
- Keeping runtime Claude OUT of per-trial key resolution. SP9a empirically tested a runtime LLM key-resolver (haiku + DOM + globals + screenshot) and found it NOT supported; SP12 removed it. The absence is an evidence-backed dead-end, not an oversight. Do not revive an in-band runtime trigger that cannot detect its own failure; the tractable lever, if per-trial correctness is ever genuinely needed, is Stage-1 source extraction validated offline. (claude-001, genbottle-003)
- The LLMClient Protocol seam (narrow 3-method, dependency-injected into every stage) and the TaskCard-as-JSON-file data contract between Reasoner and Executor at the file level. These are the clean decompositions; use them as the template when fixing the coupling found elsewhere. (arch-014)
- The well-designed robustness substrate that the new integrity gates build ON: the clear zero-trial RuntimeError with named causes, the per-stage run_trace.json with entry-nav per-phase success/error, the paradigm-agnostic tokenized withhold-sentinel, and the calibration estimator's 'never raise, always return a tagged model' contract. Do not replace these — extend them. (robust-010)
- The documented, deliberate non-claims that keep the scope honest: keyboard-only responses, point-estimate-within-range gating (not KS/Wasserstein), no semantic content, no cross-paradigm transfer, no strategic individual differences. These are sound boundary choices; the only fix needed is reconciling two internal-doc target-list inconsistencies (WCST, Sternberg) against them, not widening the model. (domgen-002, domgen-006)
