# SP1.5 — Stage 1 Prompt Gaps: Design

**Date:** 2026-05-04
**Author:** Logan Bennett (via Claude Opus 4.7)
**Status:** Approved
**Predecessor:** `2026-04-23-taskcard-reasoner-design.md` (SP1)

## Goal

Close four gaps in SP1's Stage 1 of the Reasoner that caused regenerated TaskCards to be missing fields the executor reads. Tighten the Stage 1 prompt to require the fields explicitly, add a post-stage validator that fails loudly when fields are missing, and accumulate a `reasoning_chain` across all 5 stages so the TaskCard is fully peer-reviewable.

## Background

SP1's final code review identified four gaps in the regenerated TaskCards:

1. `runtime.advance_behavior.advance_keys` was empty in 2/4 TaskCards.
2. `runtime.advance_behavior.feedback_fallback_keys` was empty in 2/4 TaskCards.
3. `runtime.data_capture.method` (and supporting `expression` / `button_selector` / `result_selector`) was missing in all 4 TaskCards. The live executor smoke ran but did not save `experiment_data.csv`.
4. `reasoning_chain` was an empty list in all 4 TaskCards. The TaskCard schema includes this field as the auditable thinking-trace from the Reasoner; no stage was populating it.

Gaps 1–3 were patched manually during SP1 from the v1 cache before deletion. Gap 4 is unaddressed. The root cause is the Stage 1 prompt does not require these fields explicitly. SP1.5 closes the loop without rebuilding the Reasoner.

## Architecture (single-section)

```
Stage 1 (existing)
  user_message includes explicit REQUIRED-fields checklist
  output passed through _validate_stage1_output()
  raises Stage1ValidationError if any required field is empty
  appends ReasoningStep("stage1_structural") to reasoning_chain accumulator

Stages 2-5 (existing) — each appends one ReasoningStep to the accumulator

Pipeline (existing)
  carries the reasoning_chain list across stages
  saves it to disk alongside each stage partial
  final TaskCard includes the full chain
```

## Component changes

### 1. Tighten the Stage 1 prompt

`src/experiment_bot/prompts/system.md` already contains schema documentation. The Stage 1 user-message builder in `src/experiment_bot/reasoner/stage1_structural.py` will be amended to append an explicit REQUIRED runtime fields checklist:

```
## REQUIRED runtime fields you MUST populate

The executor will fail or skip steps if any of these are empty. Use the
experiment's source code to determine the right values; do NOT use generic
defaults.

- `runtime.advance_behavior.advance_keys` (list of key names) — keys the bot
  presses to advance instruction or feedback screens. Examples: `[" "]` for
  jsPsych Space-advance, `["Enter"]` for ExpFactory custom HTML. Required when
  the experiment uses keypresses to advance; empty list `[]` is only acceptable
  if all advance happens via mouse clicks via `feedback_selectors`.

- `runtime.advance_behavior.feedback_fallback_keys` (list of key names) —
  fallback keys when no `feedback_selectors` button matches. Same conventions
  as `advance_keys`.

- `runtime.data_capture.method` (string: "js_expression" | "button_click" | "")
  — how the bot extracts the experiment's recorded data after completion.
  - "js_expression": provide `runtime.data_capture.expression` (a JS expression
    that returns the data as a string). Common: `jsPsych.data.get().json()` for
    jsPsych-7, `jsPsych.data.get().csv()` for csv. STOP-IT calls a custom
    `jsPsych.data.getInteractionData()`.
  - "button_click": provide `runtime.data_capture.button_selector` (a CSS
    selector for the "show data" button) and `result_selector` (selector for
    the result element). cognition.run typically uses `#data` as result.
  - "" only if the experiment has no native data save and the bot's bot_log.json
    is the only data source. Choose with caution.

- `runtime.data_capture.format` (string: "csv" | "tsv" | "json") — required
  alongside `method` when `method != ""`.
```

The Stage 1 prompt builder (`_build_stage1_prompt`) will append this section verbatim.

### 2. Validation function

New file: `src/experiment_bot/reasoner/validate.py`

```python
class Stage1ValidationError(ValueError):
    """Raised when Stage 1 output is missing executor-required fields."""


def validate_stage1_output(partial: dict) -> None:
    """Assert that Stage 1's output contains all executor-required runtime fields.

    Called after _extract_json and after normalize_partial. Raises
    Stage1ValidationError with a specific field name on the first missing
    required field. The pipeline's existing error handling preserves the
    Stage 1 partial on disk, so --resume re-runs Stage 1 with the tightened
    prompt the next invocation.
    """
    runtime = partial.get("runtime", {})
    advance = runtime.get("advance_behavior", {})

    if not advance.get("advance_keys") and not advance.get("feedback_selectors"):
        raise Stage1ValidationError(
            "runtime.advance_behavior.advance_keys is empty AND "
            "feedback_selectors is empty — the executor will be unable to "
            "advance past instruction/feedback screens. Populate at least one."
        )
    if not advance.get("feedback_fallback_keys") and not advance.get("feedback_selectors"):
        raise Stage1ValidationError(
            "runtime.advance_behavior.feedback_fallback_keys is empty AND "
            "feedback_selectors is empty — feedback screens will stall."
        )

    capture = runtime.get("data_capture", {})
    method = capture.get("method", "")
    if method == "js_expression" and not capture.get("expression"):
        raise Stage1ValidationError(
            "runtime.data_capture.method is 'js_expression' but expression is empty."
        )
    if method == "button_click" and not (capture.get("button_selector") and capture.get("result_selector")):
        raise Stage1ValidationError(
            "runtime.data_capture.method is 'button_click' but button_selector "
            "or result_selector is empty."
        )
    # method == "" is permitted but logged as a soft warning by the caller.
```

Note: `advance_keys` validation is conditional on `feedback_selectors` being empty — if Claude provides a non-empty selector list, keypress advance is unnecessary. This keeps the validator from over-fitting to keypress-driven tasks.

### 3. Reasoning chain accumulation

`ReasoningStep` already exists in `src/experiment_bot/taskcard/types.py`. Each Reasoner stage will produce one `ReasoningStep` describing what it did:

```python
# Stage 1 example
ReasoningStep(
    step="stage1_structural",
    inference="Identified Stroop paradigm from main.js. Stimuli: congruent + incongruent via stim_color/stim_word match. 2 conditions, key_press maps to z/m. Found jsPsych 7.x via plugin imports.",
    evidence_lines=["main.js:142", "experiment.js:67-89"],
    confidence="high",
)
```

Pipeline orchestration changes:

- `ReasonerPipeline.run()` initializes `reasoning_chain: list[ReasoningStep] = []` at the top.
- After each stage, the stage function returns `(partial, reasoning_step)` instead of just `partial`. The pipeline appends the step to the chain.
- The chain is included in each `stageN.json` saved partial under key `"_reasoning_chain"` (so `--resume` reads it back).
- The final TaskCard's `reasoning_chain` field is set from the accumulator.

This is a structural change to all 5 stage signatures. Each stage gets a small docstring update about what it should put in the `inference` field.

### Stage-by-stage `inference` content

| Stage | `inference` describes |
|---|---|
| 1 (structural) | Identified paradigm + how (which source files / line ranges). Listed stimuli + conditions inferred. |
| 2 (behavioral) | Per-condition mu/sigma/tau values + which study population they reflect (healthy adult, online vs lab). |
| 3 (citations) | Number of citations produced + paradigm-canonical sources used (Whelan 2008 for ex-Gaussian, Verbruggen 2019 for SSRT, etc.). |
| 4 (DOI verify) | N citations submitted to OpenAlex; M verified, K flagged as unverified-but-kept. |
| 5 (sensitivity) | Which parameters tagged "high" sensitivity and why (typically `mu`). |

## Tests

New test file: `tests/test_reasoner_validate.py`

- `test_validate_passes_on_complete_partial` — handcrafted complete partial → no exception.
- `test_validate_fails_on_missing_advance_keys_and_no_feedback_selectors` — partial with both empty → `Stage1ValidationError` with field name in message.
- `test_validate_passes_when_feedback_selectors_present_but_advance_keys_empty` — partial with `feedback_selectors=["#next"]` and `advance_keys=[]` → no exception.
- `test_validate_fails_on_missing_data_capture_expression` — `method="js_expression"` with `expression=""` → error.
- `test_validate_fails_on_missing_data_capture_button_selectors` — `method="button_click"` with empty selectors → error.
- `test_validate_passes_on_method_empty` — `method=""` is permitted (with a warning logged).

Existing test files updated:

- `tests/test_reasoner_stage1.py` — fake `LLMResponse` produces a complete partial; verify `validate_stage1_output` is called and Stage 1 returns `(partial, ReasoningStep)`.
- `tests/test_reasoner_pipeline.py` — verify the pipeline accumulates a `reasoning_chain` list of length 5 (one per stage) in the final partial.
- `tests/test_taskcard_loader.py` — round-trip a TaskCard with a non-empty `reasoning_chain` to disk and back.

## Bring-up

After tests pass:

1. Delete `.reasoner_work/` to force fresh Stage 1 against the tightened prompt: `rm -rf .reasoner_work`
2. Re-run the Reasoner on all 4 dev tasks: `bash /tmp/sp1_regen.sh` (the existing helper script).
3. For each new TaskCard, verify:
   - `runtime.advance_behavior.advance_keys` non-empty (or `feedback_selectors` non-empty)
   - `runtime.advance_behavior.feedback_fallback_keys` non-empty (or `feedback_selectors` non-empty)
   - `runtime.data_capture.method` populated, with corresponding subfields
   - `reasoning_chain` length = 5 with one entry per stage
4. Replace the 4 existing TaskCards in `taskcards/` with the regenerated ones. Delete stale earlier-attempt files.
5. Run `RUN_LIVE_LLM=1 uv run pytest tests/test_executor.py::test_live_executor_runs_against_regenerated_taskcard -v` to confirm `experiment_data.csv` is now produced.
6. Commit + tag `sp1.5-complete`.

## Out of scope

- Validators for Stages 2–5 (their outputs are checked by downstream construction).
- Auto-retry on validator failure (deferred to SP2.5 if needed).
- JSON Schema-based validation library (would replace ad-hoc validator; out of scope for v1).
- Reasoning chain UX in audit reports (SP6).

## Risks

- **Stage 1 validation may fail on the first re-run for one or more tasks.** Mitigation: the `--resume` flag preserves stages 2-5 partials; only Stage 1 re-runs. The error message names the missing field. If validator fires, refine the prompt and re-run. Loop terminates on first clean Stage 1.
- **Reasoning chain accumulator changes 5 stage signatures.** Mitigation: each change is small (return tuple instead of dict). Tests cover each stage independently.
- **The `--resume` saved partials don't yet contain reasoning chain.** Old stageN.json files will lack `_reasoning_chain` key. Pipeline reads with `.get("_reasoning_chain", [])` for backward compat.
